# Подготовка окружения на виртуальной машине
Включает в себя следующие этапы:
* Развернуть ВМ на KVM/QEMU с помощью virt-manager/virsh;
* Настроить SSH-подключение по ключу;
* Настроить файрвол.

## Виртуальная машина
Виртуальная машина имитирует реальный физический компьютер со своим процессором, памятью, жестким диском и сетевой картой, что позволяет запустить отдельную операционную систему внутри себя. 

Гипервизор представляет из себя программное обеспечение, которое берет ресурсы физического компьютера (хоста) и распределяет их между виртуальными машинами (гостями). Гипервизоры бывают двух типов. Первый устанавливается на голом железе, имеет прямой доступ к железу, что обеспечивает максимальную скорость и стабильность. Примеры: VMware ESXi, Microsoft Hyper-V, Proxmox VE. Второй тип устанавливается как обычная программа внутри пользовательской операционной системы. Может работать медленнее из-за того, что делит ресурсы с другими программами. Примеры: Oracle VirtualBox, VMware Workstation.

KVM (Kernel-based Virtual Machine) являктся модулем ядра Linux, который превращает операционную систему Linux в гипервизор 1-го типа. KVM берет на себя управление процессором и памятью, задействуя аппаратную виртуализацию процессоров Intel (VT-x) и AMD (AMD-V). QEMU (Quick Emulator) это программа, которая эмулирует все остальные компоненты компьютера: материнскую плату, видеокарту, сетевые адаптеры, жесткие диски и USB-порты. Сам по себе KVM не умеет создавать полноценный компьютер, а QEMU без KVM работает очень медленно. Когда они объединяются в стек KVM/QEMU, KVM отвечает за быструю работу процессора на уровне ядра, а QEMU создает вокруг него виртуальное «железо». По умолчанию пользовательского интерфейса у стека нет, как у VirtualBox или VMware Workstation, потому управляется через консоль или `virt-manager`. 

### Настройка Virt-Manager для QEMU/KVM на Arch
* Установка необходимых пакетов (гипервизор, утилиты управления сетью и графический менеджер)
``` bash
sudo pacman -S qemu-full virt-manager libvirt dnsmasq iptables-nft bridge-utils
```

* Настройка прав доступа, чтобы управлять виртуальными машинами без sudo
``` bash
sudo usermod -aG libvirt $USER
```

* Запуск и включение служб
``` bash
sudo systemctl enable --now libvirtd
```

* Настройка сети по умолчанию
``` bash
sudo virsh net-autostart default
sudo virsh net-start default
```

### Команды virsh для работы из консоли

* `virsh list` — Показать список только работающих виртуальных машин.
* `virsh list --all` — Показать все существующие виртуальные машины.
* `virsh start <имя_ВМ>` — Включить виртуальную машину.
* `virsh shutdown <имя_ВМ>` — Корректно выключить ВМ (отправляет сигнал ACPI).
* `virsh destroy <имя_ВМ>` — Принудительно выключить.
* `virsh reboot <имя_ВМ>` — Перезагрузить виртуальную машину.
* `virsh autostart <имя_ВМ>` — Включить автоматический запуск ВМ при загрузке ПК.
* `virsh autostart --disable <имя_ВМ>` — Отменить автозапуск машины.
* `virsh edit <имя_ВМ>` — Открыть XML-конфигурацию машины для редактирования.
* `virsh undefine <имя_ВМ>` — Удалить регистрацию ВМ из системы (сама конфигурация удалится, но файлы дисков останутся).

### Режимы сетевого адаптера

| Критерий | NAT (По умолчанию) | Bridged (Мост) | Host-only (Изолированная) |
|---|---|---|---|
| Доступ в интернет из ВМ | Да | Да | Нет |
| ВМ видна из локальной сети | Нет | Да (как отдельный ПК) | Нет |
| Где ВМ берет IP-адрес | От гипервизора (libvirt) | От роутера (DHCP) | От внутренней службы хоста |
| Безопасность для хоста | Высокая | Средняя (ВМ уязвима извне) | Максимальная |
| Когда использовать | Тестирование и обновление ПО | Внутри ВМ разворачивается сервер | Анализ вредоносного ПО, тестирование сетевых атак или работы с конфиденциальными базами данных |

## Доступ к ВМ по SSH

* Генерация пары ключей на хосте
``` bash
ssh-keygen
```
Ключевые файлы в директории `~/.ssh/`:
* `id_ed25519` — Закрытый (приватный) ключ.
* `id_ed25519.pub` — Открытый (публичный) ключ.

* Копирование открытого ключа на ВМ (предварительно узнав адрес ВМ с помощью `ip a`)
``` bash
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@192.168.122.222
```
* Файл на виртуальной машине `~/.ssh/authorized_keys` содержит список всех публичных ключей, которым разрешен вход под данным пользователем.
* Отключение авторизации по паролю на ВМ. В главном конфигурационном файле SSH-сервера `/etc/ssh/sshd_config` изменить следующие строки:
``` bash
   PubkeyAuthentication yes
   PasswordAuthentication no
   PermitEmptyPasswords no
```
* Перезапустить службу SSH внутри виртуальной машины:
``` bash
sudo systemctl restart sshd
```
   
* Удобная настройка вызова через Алиас на хост-машине через конфиг `~/.ssh/config` (пример):
``` bash
   Host myvm
       HostName 192.168.122.222
       User user
       IdentityFile ~/.ssh/id_ed25519
```

* Теперь для подключения достаточно ввести:
``` bash
ssh myvm
```

## UFW
Межсетевой экран (файрвол) — это программно-аппаратный или программный элемент компьютерной сети, осуществляющий фильтрацию, контроль и сепарацию проходящего через него сетевого трафика на основе заданных критериев и правил безопасности. Файрвол всегда делит трафик по принципу «Кто был инициатором диалога?»:
* Исходящий (Outgoing): Сервер сам проявляет инициативу. Например, при `apt update` сервер отправляет пакет-запрос в интернет.
* Входящий (Incoming): Кто-то снаружи пытается заговорить с сервером первым. Например, легитимный пользователь подключается по SSH.

Современный брандмауэр UFW в Linux — это **Stateful Firewall** (Файрвол с контролем состояний). Он не просто глухая стена, он обладает «памятью». Когда отправляется запрос наружу (например, DNS-запрос на порт 53, чтобы узнать IP-адрес сайта, или DHCP-запрос на порт 68, чтобы получить сетевой адрес от роутера), файрвол делает микро-запись в своей таблице состояний: «Мой сервер только что открыл диалог с сервером Х. Я жду ответа».Каждому пакету присваивается статус (State):
   1. `NEW` — пакет пытается открыть новое, еще не существующее соединение (кто-то ломится к нам).
   2. `ESTABLISHED` — пакет является частью уже установленного, двустороннего диалога (нам отвечают на наш же запрос).
   3. `RELATED` — пакет технически связан с уже открытым соединением.

Благодаря этому UFW автоматически пропускает любые входящие ответы (`ESTABLISHED`), если сервер был инициатором запроса. Политика запрета на вход (`deny incoming`) уничтожает только те пакеты, которые имеют статус `NEW` на незащищенных портах. Stateless Firewall (без сохранения состояния) не помнит историю пакетов. Для него каждый пакет — как первый раз. Если настроить его в режиме «запретить входящие, разрешить исходящие», интернет бы полностью сломался. В Stateless-режиме пришлось бы вручную писать небезопасные правила для обратного трафика (открывать порты 53 и 67 на вход для всех).

### Настройка файрвола только на SSH
* Проверить статус UFW:
``` bash
sudo ufw status verbose
```

* Если написано `Status: inactive`, значит файрвол выключен и пока пропускает абсолютно всё.
* Смотрим, какие порты и службы сейчас активны в системе:
``` bash
sudo ss -tulpan
```

* Могут присутствовать строки с процессами `systemd-resolve` (порт 53, локальный DNS) и `systemd-network` (порт 68, DHCP клиент).
* Сбрасываем файрвол к заводским настройкам:
``` bash
   sudo ufw reset
```
* Задаем политики по умолчанию (Выпускать всех, никого постороннего не впускать):
``` bash
   sudo ufw default deny incoming
   sudo ufw default allow outgoing
```
* Вносим SSH в белый список (разрешаем вход на порт 22):
``` bash
   sudo ufw allow 22/tcp
```
* Активируем файрвол:
``` bash
   sudo ufw enable
```

* Проверить итоговый статус:
``` bash
sudo ufw status verbose
```

* Статус изменится на `Status: active` и таблицу, где на вход (To) разрешен только порт 22/tcp для IPv4 и IPv6.
